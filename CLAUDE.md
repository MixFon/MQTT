# CLAUDE.md — IoT backend: ESP32 → MQTT → Go → TimescaleDB

Этот файл — рабочий план для Claude Code при разработке бэкенда. Держи его в актуальном
состоянии: отмечай выполненные пункты, дополняй решениями, если план меняется по ходу работы.

## О проекте

Бэкенд принимает показания с ESP32-датчиков (сейчас — температура и влажность с DHT11,
в перспективе — другие типы) по MQTT, сохраняет их в TimescaleDB и отдаёт через REST API
для визуализации (Grafana и/или собственный веб-дашборд). Схема данных и топики изначально
проектируются так, чтобы новый тип датчика добавлялся без миграций схемы БД — см. «Модель
данных» ниже.

Если в проекте уже есть Go-бэкенд (другие модули, auth, VPS-инфраструктура) — этот
функционал добавляется как отдельный пакет/модуль, без разрушения существующей структуры.
Проверяй `go.mod` и текущую структуру перед тем как создавать новые директории.

## Принцип: минимум сторонних зависимостей

Приоритет — решать задачи встроенными механизмами Go (`net/http`, `database/sql`, `log/slog`,
`encoding/json`, `context`, горутины/каналы) вместо готовых библиотек, даже если это чуть
дольше писать. Новая зависимость в `go.mod` — осознанное решение, а не выбор по умолчанию.

Добавлять стороннюю библиотеку можно только когда стандартная библиотека не решает задачу
принципиально — то есть сама область не входит в stdlib (реализация протокола, отсутствующего
в Go: MQTT, драйвер конкретной БД). В этом случае — брать максимально узкую, без транзитивных
зависимостей библиотеку, а не комбайн с лишними фичами. Роутинг, миграции, конфиг — всё это
решается в проекте без внешних пакетов, см. таблицу стека ниже.

## Архитектура

```
ESP32 (DHT11) --Wi-Fi/MQTT--> Mosquitto (VPS, TLS) --> Go: MQTT subscriber
                                                              |
                                                              v
                                                        TimescaleDB
                                                              |
                                                              v
                                                   Go: REST API (JSON)
                                                        /          \
                                                  Grafana      свой веб-дашборд
```

## Технологический стек

| Компонент          | Выбор                                  | stdlib или сторонняя |
|--------------------|-----------------------------------------|------------------------|
| MQTT broker        | Mosquitto (TLS + логин/пароль)          | внешний сервис, не Go-код |
| MQTT client (Go)   | `github.com/eclipse/paho.mqtt.golang`   | сторонняя — протокол вне stdlib, альтернативы нет |
| БД                 | PostgreSQL + расширение TimescaleDB     | — |
| Go DB driver       | `lib/pq`                                | сторонняя, но минимальная: только реализация wire-протокола Postgres поверх `database/sql`, без транзитивных зависимостей |
| Миграции           | самописный раннер (`.sql`-файлы через `embed.FS` + `database/sql`) | stdlib |
| HTTP роутер        | `net/http` + `ServeMux` (Go 1.22+, есть методы и path-параметры) | stdlib |
| Логирование        | `log/slog`                              | stdlib |
| Конфиг             | `os.Getenv` / `os.LookupEnv`            | stdlib |
| JSON               | `encoding/json`                         | stdlib |

Единственные две сторонние зависимости в проекте — MQTT-клиент и драйвер БД, и обе
неизбежны: сами протоколы (MQTT, Postgres wire protocol) не входят в стандартную библиотеку
Go. Всё остальное — только stdlib.

## Модель данных

Схема — "narrow" (одна строка на метрику, а не колонка на метрику). Выбрана осознанно,
т.к. планируется добавление новых типов датчиков позже: новый тип метрики — это просто
новое значение `metric`, без `ALTER TABLE` и без передеплоя.

```sql
CREATE TABLE sensor_readings (
    time   TIMESTAMPTZ NOT NULL,
    room   TEXT NOT NULL,
    metric TEXT NOT NULL,   -- 'temperature', 'humidity', в будущем: 'co2', 'lux', ...
    value  REAL NOT NULL
);
SELECT create_hypertable('sensor_readings', 'time');
CREATE INDEX ON sensor_readings (room, metric, time DESC);
```

Компромисс этого выбора: чтение "температура и влажность рядом в одной строке" требует
`GROUP BY time, room` со сведением через `MAX(value) FILTER (WHERE metric = 'temperature')`
(или два отдельных запроса на фронте) вместо простого `SELECT temperature, humidity`.
Это стандартная и хорошо изученная плата за гибкость в time-series (тот же подход у
Prometheus/InfluxDB) — принимаем её ради того, чтобы не переделывать схему при каждом
новом датчике.

## MQTT: соглашение по топикам

```
home/{room}/{metric}   payload: числовое значение, например "23.4"
home/{room}/status     payload: "online" / "offline"  (через Last Will)
```

`{metric}` — не фиксированный список, а открытое множество: `temperature`, `humidity`,
позже `co2`, `lux` и т.д. Подписчик слушает `home/+/+` и определяет комнату и метрику
из самого топика — новый тип датчика не требует правок в коде подписчика, только
прошивку ESP32, публикующую в новый топик.

QoS: 1 (at-least-once) достаточно для показаний датчика, дублирование не критично.

`.env` — только для локальной разработки, в `.gitignore`. На VPS — через systemd
`Environment=` или отдельный env-файл вне репозитория (тот же подход, что уже
использовался для credentials в проекте ESP32).

## Соглашения по коду

- Комментарии в коде — на русском языке
- У каждой функции и метода, которые пишет Claude, должен быть комментарий,
  описывающий, что она делает (не только у экспортируемых — у всех)
- Ошибки оборачивать через `fmt.Errorf("...: %w", err)`, не глотать молча
- Один пакет — одна зона ответственности (`mqtt` не знает про HTTP, `api` не знает про MQTT)
- Конфиг читается один раз в `main.go` и передаётся явно (dependency injection через конструкторы, без глобальных переменных)
- Коммиты — по одному законченному шагу из плана выше, сообщения на английском в императиве (`add mqtt subscriber`, `wire timescaledb schema`)
- Перед добавлением новой зависимости в `go.mod` — сначала проверить, решается ли задача
  стандартной библиотекой. Если да — писать самому, даже если это на 20-30 строк больше кода.
  Список допустимых исключений закрыт и ограничен таблицей стека выше (MQTT-клиент, драйвер БД)

## Открытые вопросы

- Один датчик или несколько комнат сразу? (влияет на топики и на UI — но схема уже готова к обоим случаям)
- Нужна ли аутентификация на REST API, или он закрыт VPN/firewall?
- Нужен ли WebSocket/SSE для live-обновления дашборда, или polling раз в N секунд достаточен?

Решено: планируются датчики, отличные от DHT11 (не только температура/влажность) — поэтому
схема БД и топики спроектированы как открытое множество метрик, см. «Модель данных» и
«MQTT: соглашение по топикам».

## Вне текущего плана

Свой MQTT-брокер на замену Mosquitto — сознательно не часть этого плана. Mosquitto остаётся
основным брокером в проде. Если позже появится желание написать брокер вручную (используя
`net` + горутины + каналы вместо `paho`) — делать это как отдельный побочный проект, гонять
параллельно с Mosquitto на другом порту, и переключать `MQTT_BROKER_URL` только после того,
как реализация проверена на реальном трафике с ESP32 и не теряет/не дублирует сообщения.
