# План: скетч ESP32 → Mosquitto по MQTT (PlatformIO + VSCode)

## Контекст

Есть работающий бэкенд: Mosquitto на VPS (`mrmixfon.ru:8883`), TLS с самоподписанным CA, аутентификация по логину/паролю. Топики:

```
home/{room}/{metric}   payload: числовое значение как строка, например "23.4"
home/{room}/status     payload: "online" / "offline" — публикуется через MQTT Last Will
```

- `{room}` и `{metric}` — открытое множество, не фиксированный список (например `room=bedroom`, `metric=temperature`/`humidity`).
- QoS: 1 (at-least-once).
- Сертификат брокера — самоподписанный, выпущен на CN/SAN `mrmixfon.ru`. Подключаться нужно по этому домену, не по IP — иначе проверка SAN на стороне клиента не пройдёт.
- CA-сертификат (`ca.crt`), которым подписан серверный сертификат, нужно зашить в прошивку — устройство пинит конкретный CA, а не проверяет системный список доверенных корней (на ESP32 его и нет).

## Этап 1 — окружение PlatformIO

- [ ] Инициализировать проект PlatformIO под плату ESP32 (`platform = espressif32`, `board = <конкретная плата, уточнить у пользователя>`, `framework = arduino`)
- [ ] Настроить `platformio.ini`: `monitor_speed`, `upload_speed`, при необходимости `upload_port`/`monitor_port`
- [ ] Выбрать MQTT-клиент: `knolleary/PubSubClient` (простой, но с ограничением на размер сообщения по умолчанию — обычно достаточно для коротких payload) или `256dpi/MQTT` (более гибкий API). Указать в `lib_deps`
- [ ] Для TLS использовать встроенный `WiFiClientSecure` (часть `espressif32` framework, не требует отдельной зависимости)

## Этап 2 — хранение секретов

- [ ] Wi-Fi SSID/пароль, MQTT логин/пароль и CA-сертификат — не хардкодить в основном `.cpp`-файле и не коммитить
- [ ] Завести `include/secrets.h`, добавить его в `.gitignore`; закоммитить только `include/secrets.h.example` с плейсхолдерами
- [ ] В `secrets.h` хранить: `WIFI_SSID`, `WIFI_PASSWORD`, `MQTT_HOST` (`"mrmixfon.ru"`), `MQTT_PORT` (`8883`), `MQTT_USERNAME`, `MQTT_PASSWORD`, `ROOM` (имя комнаты для этого устройства) и сам CA-сертификат как `const char* MQTT_CA_CERT PROGMEM = "-----BEGIN CERTIFICATE-----\n...";`
- [ ] Явно проговорить в плане: CA-сертификат для прошивки — это `ca.crt`, сгенерированный на сервере (см. `deploy/VPS_DEPLOY.md`, раздел 3, или локально через `deploy/mosquitto/gen-dev-certs.sh` для тестового стенда). Не путать с `server.crt`/`server.key` — они на устройство не идут, это приватные ключи брокера

## Этап 3 — Wi-Fi и синхронизация времени

- [ ] Подключение к Wi-Fi (`WiFi.begin`), ожидание `WL_CONNECTED` с таймаутом и логом в Serial
- [ ] **Важный нюанс, часто пропускают**: `WiFiClientSecure::setCACert` проверяет срок действия сертификата, а у ESP32 после старта время не установлено (эпоха 1970) — проверка сертификата будет падать с ошибкой рукопожатия, даже если сам сертификат и CA корректны. Поэтому после подключения к Wi-Fi обязательно синхронизировать время через NTP (`configTime` + ожидание, пока `time(nullptr)` не станет реалистичным), и только после этого пытаться подключаться к MQTT

## Этап 4 — TLS + MQTT подключение

- [ ] Создать `WiFiClientSecure`, вызвать `setCACert(MQTT_CA_CERT)`
- [ ] Обернуть в `PubSubClient` (или выбранный клиент), настроить `setServer(MQTT_HOST, MQTT_PORT)`
- [ ] Настроить Last Will перед подключением: топик `home/{room}/status`, payload `"offline"`, retain — обсудить с пользователем, нужен ли retain (обычно да, чтобы новый подписчик сразу видел последний известный статус)
- [ ] После успешного `connect()` с логином/паролем — опубликовать `home/{room}/status` = `"online"` (retain, если решили выше)
- [ ] Подключение с логином/паролем — брокер настроен с `allow_anonymous false`

## Этап 5 — публикация показаний

- [ ] Опрос датчика (DHT11 или другой — уточнить у пользователя, какая библиотека для конкретного датчика)
- [ ] Публикация в `home/{room}/temperature` и `home/{room}/humidity` (или другие метрики) — payload как строка числа (`String(value, 1)` или `dtostrf`), QoS 1
- [ ] Интервал публикации — таймер на `millis()`, не `delay()` (чтобы не блокировать `client.loop()` и обработку MQTT keep-alive)
- [ ] Если значение с датчика невалидно (NaN, ошибка чтения) — не публиковать, залогировать в Serial и повторить на следующем цикле (бэкенд не делает валидацию на приёме — мусор в БД лучше не пускать на уровне прошивки)

## Этап 6 — устойчивость соединения

- [ ] Цикл переподключения к MQTT при обрыве (`client.connected()` проверка в `loop()`, реконнект с backoff, а не в жёстком цикле — не долбить брокер)
- [ ] Отдельно обработать обрыв Wi-Fi (переподключение к сети до попытки переподключения к MQTT)
- [ ] `client.loop()` вызывать на каждой итерации `loop()`, без длинных блокирующих `delay()`

## Этап 7 — проверка

- [ ] Сначала протестировать на локальном тестовом брокере (Mosquitto из `docker-compose.yml` этого проекта + сертификаты из `deploy/mosquitto/gen-dev-certs.sh`) — там же логин/пароль `iot`/`changeme` по умолчанию
- [ ] Проверить получение данных: `mosquitto_sub -h <host> -p 8883 --cafile ca.crt -u iot -P <пароль> -t 'home/#' -v`
- [ ] После локальной проверки — переключить `secrets.h` на боевые значения (`mrmixfon.ru`, боевой CA, боевой пароль) и проверить на реальном брокере
- [ ] Убедиться, что показания доходят до бэкенда: `GET https://mrmixfon.ru/api/latest?room={room}` должен вернуть свежие данные
